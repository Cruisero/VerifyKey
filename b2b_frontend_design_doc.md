# Gemini 分销版独立验证工作台 - 全栈开发指引

## 📖 0. 业务全景图与架构概念（研发人员必读）

如果你是刚接手这个项目的开发，你需要知道本项目的业务逻辑是分级的。不要把它当成一个孤立的前端站，它实际上是一个涵盖了**独立管理后台**和**下游验证面板**的小型分销枢纽。

**你的角色与全景逻辑**：
我们（你的甲方，也就是主线系统）拥有一套最底层的、类似 UPixel 一样的强大核芯自动化验证引擎（名为 **OnePass API**）。
你现在要独立开发一个全新的系统，对下（客户）充当“代刷平台”，对上（向我们）充当“大代理商”。

**业务运转闭环**：
1. **上游配置**：你的后台有个设置页面，你需要把你向我们采购的「上游大额 API Key / 上游总卡密」以及上游请求域名填进去。
2. **下游制卡**：你的后台需要自己一套生成、管理小额度卡密（**CDK**）的功能，这就是你在你前端变现、分发出去的凭证。
3. **客户使用**：B 端客户打开你开发的前端网页（纯工具），不存在登录面板。输入你生成的 CDK，进行大批量的验证链接代刷。
4. **流转扣费**：前端调用你的后台 -> 你的后台查验这人的小额 CDK 够不够，如果够就扣钱 -> 后台悄悄带上你的「上游大额总卡密」，转发请求呼叫我们底层的 OnePass API 执行任务 -> 我们的服务器边跑边吐进度 -> 你的后台透明返回（透传）接收进度呈现给你的前端。

## 🏗 1. 系统架构分层推荐

既然包含独立后端需求，推荐使用 **Node.js (Express 或 NestJS)** 来支持轻快的高并发透传，搭配 **Vue/React**。如果希望极其轻量，也可以单仓库使用 **Next.js** 配套内置 API Routes + SQLite 数据库。

### 👉 子系统 A：面向客户的前端站 (无账号版)
- 纯操作面板：只有输入卡密、看进度表的功能。

### 👉 子系统 B：核心分销后台 (Reseller Backend)
- **API 代理层**：对外接收前端请求，对内封装并重定向至我们的底层 API。
- **自营业务引擎**：负责管理自己发行卡密的生命周期。

## 🎯 2. 后台开发核心指标（你需要在服务端实现的模块）

### 2.1 基础的后台管理模块
- **系统设置**：用来维护两项关键信息，`Upstream_API_URL` (也就是我们的底座 API 域名，通常需要保密，不要暴露到前端) 以及 `Upstream_API_Secret`。
- **CDK 生成与管理表**：开发一个简单的界面供运营批量生成卡密。主要字段涉及 `唯一密钥 code`，`总额度`，`已耗用额度`，`过期时间`，`启用/封禁状态`。

### 2.2 请求鉴定与分发过桥（极其重要！）
前端界面绝对不可以直连我们的底层主 API（否则上游大额总秘钥全泄露了）。前端必须请求你的本地接口，例如：
`POST {你的域名}/v1/local-api/verify/distribute`

**你的后端的该接口逻辑必须如下**：
1. 提取前端穿过来的 `local_cdk` 参数。在你的数据库查实该用户 CDK 是否存在且有足够积分。（并发注意死锁处理）
2. 将发起的连接封装，在体部置换成上游要求的凭证：
    ```javascript
    // 转发组装逻辑示例 (伪代码):
    const upstreamPayload = {
       "links": request.body.links, 
       "mode": request.body.mode, // 客户选的全自动还是半自动
       "cdk": "这是你在系统设置里填写的我们发给你的大额秘钥总成"
    };
    ```
3. 用 HTTP Client 向我们的 `https://<我们的主域>/api/verify/dualbot` 发起真实请求。
4. **【核心难点：流式透传】** 我们后端吐回来的是 `Server-Sent Events (SSE)` 持续文字流，请让你的 Node.js 将接收到的数据流使用 `Response.pipe(res)` 或等同机制，无缝转推（透传）给你的前端。千万不能等我们跑全闭环（通常需几分钟）才把响应结果发给前端，不然前端界面会全部卡死超时。

## 🎨 3. 客户级打单前端规范

这是一个极简冷酷风（深色黑金主题）的工作大屏。主要满足“绑卡、投喂链接、查实时录屏进度”三项交互。

### 3.1 顶部全局状态栏
*   不显示任何个人中心图标。最右边常驻展示功能区：**卡密输入框** 及 **[刷新/验证] 按钮**。
*   卡密验证通过后，前端调用你实现的 `GET /local-api/cdk/info`，在这里永远点亮并显示当前该卡还有多少残余额度。

### 3.2 模式选择与批量发包口
*   提供一个显眼的【添加链接】抽屉弹窗支持多文本域导入 `textarea`。（你需要加正则只提取 `services.sheerid.com/verify` 开头的干净 URL）
*   **【关键配置】：加入「半自动」和「全自动」任务开关 Switch。**
    - 全自动，传递 `mode: 'auto'`。底层 API 直接做到最后过白。
    - 半自动，传递 `mode: 'semi-auto'`。底层 API 提供合成的头像和证件材料然后终止。你需要捕捉后端透传的数据做个附件相册呈现。

### 3.3 无限滚动的跑马大屏幕 (数据明细呈现处)
*   分为四个状态标签卡（Total 全部, Pending 执行中, Success 成功, Failed 失败报红）。
*   你需要将发去后端的 `links` 解析开，形成大表格里的数十行 Row。
*   因为 SSE 事件是在后台源源不断传来，前端不要每个链接单发网络去查。而是利用流返回的每个单独 `vid: 1234xxx` 带着 `message` 块事件去找到前端状态管理池 (Redux/Pinia) 中的同一行记录单元，响应式推入并重绘 `状态` 和 `当前详情 Message` 两个表格列，形成真实流畅的 AI打单文字输出动态效果。
*   **单机并发拦截器限制**：前端向后的批量发送要切割为轮替的 Batch Array，绝不可一次传入>5条以上的 Links 去挤压你的透传服务器，5条生命周期完结（无论好坏）再压入下 5 条。
